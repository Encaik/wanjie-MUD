## Context

基于新属性/核心值系统，实现事件驱动的双人战斗引擎。消费核心值，不依赖旧属性系统。

本次重构核心变更：**从线性 ATB tick 模型改为事件驱动衰减+保底模型**。移除全局帧/tick 概念，战斗直接由行动事件构成，通过速度衰减和保底计数器控制行动顺序。

## Goals / Non-Goals

**Goals:** 事件驱动战斗 + 速度衰减出手权 + 保底机制 + 物攻/特攻分离 + 技能CD + 多种开场类型

**Non-Goals:** 不实现战斗UI、不接入功法系统（仅预留接口）、不做多人战斗

## Decisions

### 战斗开场类型

| 类型 | 速度修正 | 说明 |
|------|----------|------|
| encounter | 无 | 正常遭遇 |
| ambush | 攻击方首轮强制先手 | 伏击 |
| surprise | 攻击方速度×1.5 | 出其不意 |
| defense | 防御方物防+50%首轮 | 防守战 |

### 核心机制：速度衰减 + 保底

**不要 tick 概念。** 战斗是纯事件驱动的——每次循环就是一次行动。

```
参数:
  DECAY = 0.5         （每次行动后速度衰减系数）
  PITY_MAX = 4        （连续行动次数上限，触发强制切换）
  ACTION_CAP = 50     （总行动次数上限，防止死循环）

每轮循环:
  1. 保底检查：同一方连续行动 ≥ PITY_MAX → 强制给对方
  2. 正常比较：比较当前有效 speed，高者获得本次行动权
  3. 行动方 speed *= DECAY
  4. 对方行动后，行动方 speed 恢复为原始值
  5. 一方 HP=0 或达到 ACTION_CAP → 结束
```

**为什么这个模型能应对 speed 膨胀：**

```
speed 10 vs 5:
  A(10)→衰减5→打平→给B → B动→A恢复10
  结果: ABABAB 基本交替，A偶尔多抢一次 ✓

speed 100k vs 10k:
  A(100k)→50k→25k→12.5k→6.25k（连动4次）
  衰减到低于10k自然切换，或 PITY_MAX=4 触发保底
  结果: AAAAB AAAAB ✓

speed 100k vs 50k:
  A(100k)→50k→打平→给B → B动→A恢复100k
  结果: ABAB 中A更频繁连抢，比10:5差异更明显 ✓
```

### 伤害公式 (宝可梦式)

```
Damage = floor(((2×level + 10) / 250 × (ATK / DEF) × power + 2) × random(0.85, 1.0))

- 物攻: ATK = physicalATK, DEF = physicalDEF
- 特攻: ATK = specialATK, DEF = specialDEF
- 暴击: 5% 概率 ×1.5
```

### 技能冷却系统

冷却改为**按自身行动次数计算**，与速度完全脱钩：

```
CD = N → 使用后需经过 N 次自身行动才能再次使用

每次自身行动时: currentCooldown = max(0, currentCooldown - 1)
```

理由：速度越快出手越多→技能自然转得快，不需要速度参与CD计算。消除原 `decay = speed/20` 的尺度问题。

### 战斗模式

- **手动**: 用户每回合选择技能（CombatSession）
- **自动**: 自动选择冷却完毕中威力最高的技能（executeCombat）

### 行动次数上限

当总行动次数达到 ACTION_CAP 时，按剩余血量比例判定胜负：
- 血量比例高者获胜
- 血量比例相同则平局
