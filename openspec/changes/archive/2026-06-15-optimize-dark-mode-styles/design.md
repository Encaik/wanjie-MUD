## Context

当前项目有完善的主题架构：CSS 变量定义在 `src/app/styles/themes.css`，通过 `:root`（亮色）和 `.dark`（暗色）两个选择器分别赋不同值。`tokens.css` 通过 `@theme inline` 将 CSS 变量桥接到 Tailwind 语义类名。`modules/theme/` 负责主题偏好管理和运行时世界主题注入。

暗色模式的变量采用 OKLCH 颜色空间，色相集中在 55-70（暖黄/棕），旨在营造"墨色古籍"氛围。但当前取值存在层次扁平、对比度不足的问题。

项目使用 Tailwind CSS v4 + `@custom-variant dark (&:is(.dark *))` 来启用 `dark:` 前缀变体。

## Goals / Non-Goals

**Goals:**
- 暗色模式下各语义层之间的视觉层次清晰可辨（背景/卡片/弹出层至少有 6-8% 的明度梯度）
- 次级/辅助文本在暗色背景下满足 WCAG AA 对比度（正文 ≥ 4.5:1，大文本 ≥ 3:1，目标 ≥ 7:1 以补偿 CJK 小字号）
- 边框在暗色背景下清晰可见
- 品质颜色和游戏领域色在暗色背景下不过亮刺眼、不过暗褪色
- 保留修仙题材"墨色古籍，烛光阅读"的暖暗调性（不对标纯黑科技风或高饱和霓虹风）
- 保留所有语义变量名不变，零破坏性

**Non-Goals:**
- 不在此变更中迁移硬编码色盘的组件到语义色（那是 `ui-theme-compliance` 的范围）
- 不改变亮色模式的任何值
- 不改变主题切换逻辑、ThemeProvider、localStorage 持久化
- 不改变 tokens.css 的桥接结构
- 不新增或删除 CSS 变量
- 不改变世界主题的运行时注入机制（`setProperty`）

## Decisions

### Decision 1: 设计目标对比度 — 正文 ≥ 7:1，辅助文本 ≥ 4.5:1

**选择**: 暗色正文（`--foreground`）与背景（`--background`）的 WCAG 对比度 ≥ 10:1，次要文本（`--muted-foreground`）与背景 ≥ 6:1。

**理由**: CJK 字符笔画密集（宋体尤甚），小字号（14px 以下）时需要更高对比度补偿。WCAG AA 的 4.5:1 是针对拉丁字母的，CJK 需要上浮 1-2 个点。选择 6:1+ 作为辅助文本最低目标是折衷——既不过亮破坏暗色氛围，又确保可读。

### Decision 2: 建立三层明度梯度（背景 → 卡片 → 弹出层）

**选择**:
```
背景层（最深）    --background:  oklch(0.16 0.015 60)   L=0.16
卡片层（中等）    --card:        oklch(0.24 0.03 60)    L=0.24  (+8%)
弹出层（最浅）    --popover:     oklch(0.32 0.04 60)    L=0.32  (+8%)
```

**替代方案**: 保持当前 L 值（0.18/0.22/0.22），仅增加色度——但测色表明增加色度不足以产生层次感，明度差是层次感的主要来源。

**理由**: 亮色模式下三层明度差为 3-5%（0.99/0.96/0.94），暗色模式下由于人眼对暗区的明度变化不敏感（Stevens 幂律），需要更大的绝对明度差才能产生相同的感知区分。8% 的明度差在 OKLCH L=0.15-0.35 区间大约等于亮色模式下 2-3% 的感知差。

### Decision 3: 文本明度分级

**选择**:
```
正文（最重要）     --foreground:            oklch(0.91 0.02 65)   L=0.91
次要文本           --secondary-foreground:  oklch(0.78 0.03 65)   L=0.78  (-13%)
辅助文本（最弱）   --muted-foreground:      oklch(0.70 0.04 65)   L=0.70  (-8%)
```

**替代方案**: 直接将 `--muted-foreground` 提到 L=0.75，但这会缩小与 `--secondary-foreground` 的差距。选择 L=0.70 是基于"比当前 0.65 明显提升，但仍与次要文本保持层次"的折衷。

**理由**: 正文与背景的 APCA 对比度约 18:1（极好），次要文本约 12:1（良好），辅助文本约 8:1（满足 CJK 可读性要求）。明度等差分布（-13%/-8%）确保文本三级层次清晰。

### Decision 4: 边框明度提升至 L=0.38

**选择**: `--border: oklch(0.38 0.03 60)`，比背景 L=0.16 高出 22 个百分点。

**理由**: 亮色模式下 `--border: oklch(0.88 0.02 70)` 与背景 L=0.96 差 8 个百分点即清晰可见。但暗色模式下同样 8% 差（L=0.24）完全不可见。由于暗区对比感知的非线性，边框需要至少 20% 以上的明度差才能达到亮色模式下的同等可见度。L=0.38 比卡片（L=0.24）亮 14%，确保卡片边框清晰可见。

### Decision 5: 增加全局色度营造"温度感"

**选择**: 暗色模式下所有基础语义色的色度从当前 0.02-0.04 整体提升至 0.03-0.06。

**理由**: 当前暗色模式色度极低（接近纯灰），给人"死黑"感。微增色度至 0.03-0.05，配合暖黄色相（hue=60-65），营造"烛光下阅读泛黄古籍"的温暖暗色感——而非"伸手不见五指的黑牢"。色度控制在 0.05 以下避免像 Safari 阅读器暗色模式那样过于暖黄失真。

### Decision 6: 品质颜色和游戏领域色微调

**选择**: 品质颜色在暗色下亮度微升（L +0.02 ~ +0.05），色度微降（chroma -0.02 ~ -0.05），避免暗色背景上高饱和颜色产生"霓虹刺眼"效应。游戏领域色同理。

**尤其关注**:
- `quality-epic`（黄色）在暗色下色度从 0.15 降至 0.12，避免黄底黑字刺眼
- `quality-common`（绿色）明度从 0.70 降至 0.68，色度从 0.12 降至 0.10
- `game-cultivation`（修炼-蓝紫色）色相从 260 调至 265，偏紫更符合"修仙"想象

### Decision 7: 基础字号确保

**选择**: 在 `base.css` body 样式中确保 `font-size: 1rem`（16px），同时添加 `-webkit-font-smoothing: antialiased` 改善暗色模式下字体渲染。

**理由**: 虽然浏览器默认已经是 16px，但显式设置确保不因 CSS reset 或框架默认值变为更小。暗色模式下的 CJK 宋体渲染在低对比度 + 无抗锯齿时笔画粘连尤其严重。

## Risks / Trade-offs

- **[风险] 暗色变亮可能打破用户预期**: 当前习惯暗色模式的用户可能觉得"变亮了不像暗色模式"。通过保留背景 L 在 0.16（仍然非常暗）缓解，变动的只是层次间的关系而非整体亮度。
- **[风险] WCAG 精确对比度未逐色验证**: OKLCH → sRGB → WCAG 对比度的转换在纸上只是近似。实施后将用浏览器 DevTools 实测。
- **[取舍] 暖黄色相 vs 色准**: 增加色度后可能在某些显示器上偏黄。选择 hue=60（比目前 55 稍偏黄）是为了温暖感，但色度 0.03-0.04 足够低，不至于在普通 sRGB 屏幕上变成明显的黄底。
- **[取舍] 此变更只修语义变量，不修硬编码色盘**: 约 64 个文件仍使用 Tailwind 原生色盘（如 `dark:text-emerald-400`），它们的暗色效果不受此变更影响。但 `themes.css` 变量更新后，所有使用语义色的组件自动受益，这是投入产出比最高的方案。
